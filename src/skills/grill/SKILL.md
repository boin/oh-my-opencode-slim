# Grill

## Purpose

Convert a vague request into a defensible spec. The orchestrator interrogates
its own assumptions first; only unanswerable or judgment-call questions are
escalated to the human (via `/interview` if available, or plain Q&A).

Output: `docs/spec/requirements.md` and `docs/spec/design.md` populated and
mutually consistent. `trace.md` is generated by `trace_regenerate` after both
exist.

## When to use

- New non-trivial feature or refactor and `docs/spec/requirements.md` is
  missing.
- User asks "design X" or "let's plan Y" without an existing spec.
- Orchestrator hits step 1 of the SDD workflow and finds no triad.

## When NOT to use

- A spec triad already exists → run `trace_regenerate` instead.
- Trivial change (typo, one-line fix, single bounded edit < 20 lines).
- Pure execution work where the user explicitly skips spec ("just do it").

## The four phases

### 1. Assumptions

List every assumption you would otherwise make silently. Format each as
`A-N: <one-sentence statement>`. Keep going until you cannot think of more.
Stop when assumptions become trivial.

### 2. Risks

For each assumption, ask: "What breaks if this is wrong?" Format as
`R-N: <consequence>`. Drop assumptions whose breakage is harmless.

### 3. Self-answerable questions

Questions the agent itself can answer by reading code, running searches, or
fetching docs. Format as `Q-N: <question> → resolved by @explorer|@librarian|self`.
Resolve them now. Inject answers as `A-N: <answer>` inline.

### 4. Human-decision questions

What remains after phase 3. These are judgment calls, scope choices, business
priorities — never things the agent could derive. Format as
`H-N: <question> [options: a | b | c]`. Route via `/interview` if the slash
command is installed; otherwise ask the user inline as a numbered list.

## External-dependency double check

For every external dependency mentioned (MCP server, third-party API, package,
shared service), the grill MUST raise both:

- **Read responsibility**: who reads from it, when, what data, why.
- **Write responsibility**: who writes to it, when, what data, with what
  guard rails (idempotency, retries, schema validation).

If either is undefined the dependency is not yet specified. Add a `H-N`
question. Do not let the spec converge without both sides answered.

## Convergence rule

- Each round = one pass through phases 1-4.
- After each round, ask: "Does the residual set of `H-N` questions block
  writing requirements?" If no → write `requirements.md` and `design.md`.
- Hard cap: **3 rounds**. If still divergent after round 3, halt and report
  to the user with the full residual question list. Do not attempt to write
  a spec on shaky ground.

## Output format

### requirements.md

```
# Requirements

## REQ-001: <short title>

<one paragraph of intent and constraints>

## REQ-002: ...
```

Each REQ is a numbered heading with a stable ID. IDs never change after
creation; new requirements append.

### design.md

```
# Design

## DES-001: <short title>

Rationale anchor: REQ-001, REQ-003.

<implementation approach in 1-3 short paragraphs>
```

The `Rationale anchor:` line is mandatory and is what `trace_regenerate`
parses. Anchor every DES to ≥1 REQ.

### Heading format is strict

`trace_regenerate` parses headings with the exact pattern
`## REQ-<digits>:` and `## DES-<digits>:` (level-2 heading, ID, then a
colon). If you write `## REQ-1 多平台插件库` without the colon, the
parser finds zero IDs and `trace.md` ends up with only the table header.
Likewise, multiple REQ anchors must be comma-separated on a single
`Rationale anchor:` line (e.g. `Rationale anchor: REQ-001, REQ-003.`) —
multi-line anchor blocks are not parsed.

If `trace.md` regenerates to an empty table, the first thing to check is
whether every REQ/DES heading has the trailing colon.

### After both files exist

Run `trace_regenerate` (the tool, not check-only). This emits `trace.md`
mapping each REQ to its DES list.

## Delta workflow (after trunk triad exists)

The above bootstrap path applies ONLY when `docs/spec/requirements.md` does
not exist yet. Once trunk is in place, every subsequent grill MUST go
through the change-proposal pipeline instead of editing trunk in place:

1. Pick a kebab-case `<slug>` for the change.
2. Call `spec_propose` with `slug` + one-line summary. The tool allocates
   the next REQ/DES IDs (accounting for other open changes) and writes
   skeletons under `docs/spec/changes/<slug>/`:
   - `proposal.md` — the one-line summary
   - `delta-requirements.md` — `## REQ-NNN: <fill in>` stub
   - `delta-design.md` — `## DES-NNN: <fill in>` stub with the
     `Rationale anchor:` line pre-stamped
3. Run the four-phase grill against the delta files only. Trunk
   `requirements.md` / `design.md` are read-only during this phase.
4. When grill converges, send the delta to `@oracle` for output review.
5. Oracle on approval calls:
   - `spec_merge slug=<slug>` — appends delta sections into trunk and
     regenerates `trace.md`. Refuses on ID collision.
   - `spec_archive slug=<slug>` — moves
     `docs/spec/changes/<slug>/` into
     `docs/spec/archive/YYYY-MM-DD-<slug>/`. Atomic rename.

Edits to existing REQs/DESs are out of scope in v1. If a delta needs to
modify an existing REQ, halt grill and flag the user.

### Slug naming

- Each open `<slug>` lives in `docs/spec/changes/<slug>/` and must be
  unique among open changes.
- After archive, the same slug can be reused on a different day (archive
  paths are date-prefixed so collisions are impossible across days).
- Same-day reuse of an already-archived slug is refused — pick a
  different slug or wait until the next day.

## Memory hooks

Before starting grill, call `recall_memories` with the project tag and a
brief topic query. Inject the top 3-5 results into your phase-1 assumptions
list as `A-prior-N: <lesson from past work>`. This is where @oracle's
pitfall/pattern memories pay back.

After convergence, do NOT save memories yourself. @oracle is the sole writer
and will save at output review (per `MEMEX_CONTRACT` in the orchestrator
prompt).

## Anti-patterns

- Skipping phase 1 because "the request is obvious" → there are always
  hidden assumptions; surface them.
- Asking the human in phase 4 things you could have answered in phase 3.
- Before writing any `H-N`, run this self-check: *"Could `grep`, a file
  read, or a doc fetch answer this?"* If yes, it belongs in phase 3,
  not phase 4. Default to resolving the question yourself; escalate
  only what is genuinely a judgment call (business priority, scope
  tradeoff, taste — never something derivable from code or docs).
- Letting an external dependency through without read+write responsibilities
  defined.
- Writing `requirements.md` while questions still block.
- More than 3 rounds. Past 3 rounds the issue is structural; escalate, do
  not grind.
